# Meta-Agent · 评比汇报材料

> **定位**：不是又一个编码助手，而是一个**可累积、可自治、可协作的工程智能体运行时**。通用编码代理（Codex / Claude Code）解决"把代码写到能跑"，meta-agent 解决"工程能力如何不退反增、长周期任务如何安全自治、跨系统全链路如何一句话驱动"。
>
> 汇报时长 ≤ 10 min · 全文对应 10 页幻灯片

---

## 第 1 页 · 封面与一句话定位

**Meta-Agent：面向工程问题的自主智能体运行时**

- **是什么**：一套自研 TypeScript 运行时（npm 包 `@meta-agent/runtime` v0.5.1），把流式模型调用、多轮工具循环、会话状态与恢复、权限沙箱、上下文压缩、自治执行、并发子代理、工程验证与知识沉淀封装成统一接口。既是库，也是开箱即用的 CLI。
- **一句话价值**：把工程代理从"一次性的临场发挥"变成**"可累积、可追溯、不退化的工程能力"**——同一个项目越做越懂，经验在累积，原则在固化，而人始终保有最终裁决权。
- **当前状态**：v0.5.1 已发 npm 包，83,000 行 TS 源码 + 18,000 行测试，60 次提交，2 名开发者，约 10 周开发周期。

## 第 2 页 · 为什么已有 Codex / Claude Code，还要开发 Meta-Agent？

### 通用编码代理的天花板

Codex、Claude Code 这类代理很强——**擅长"把代码写到能跑"**。但工程系统开发的真正难点不在写代码，而在四件事：

| 工程现实 | 通用编码代理的表现 | 后果 |
| --- | --- | --- |
| **物理世界会反复证伪你的假设** | 模型会顺着漂亮的逻辑链，把硬件约束当成可忽略的细节 | 仿真很美，上机就炸 |
| **一条经验要经过多次验证才值得信任** | 经验消散在聊天历史里，每次从零开始 | 同一个坑反复踩 |
| **错误结论一旦沉淀会污染所有后续决策** | 无知识质量管控，错误信息自由传播 | 越用越糊涂 |
| **真正的优化是长周期、多目标的** | 一次性会话，没有结构化的实验流程 | 要么失控跑偏，要么全靠人工 |

### Meta-Agent 的回答：四套相互咬合的机制

| 机制 | 对标通用代理的短板 | 核心思想 |
| --- | --- | --- |
| **Experience 经验库** | 经验不可复用 | 每次踩坑→带置信分层的结构化经验→自动召回→反复验证后晋升为原则 |
| **Physical Anchor 物理锚点** | 模型会"推理掉现实" | 钉死物理/硬件事实，模型不得擅自推翻 |
| **人工 Review** | 知识会被污染 | AI 只负责高召回提议，人负责高精度裁决，**永不自动写入共享库** |
| **Campaign 多目标优化** | 长周期任务会失控 | DOE 采样→多保真度筛选→Pareto 前沿→人工检查点 |

> **结论**：通用编码代理是"用完即弃的代码生成器"；meta-agent 是"在同一项目上越用越准、在长周期任务上安全自治、知识质量有保障的工程伙伴"。

## 第 3 页 · Meta-Agent 能解决什么业务问题

### 不止"写代码"——工程开发的全链路痛点

工程团队的核心瓶颈，远不止"代码写得快不快"。Meta-Agent 覆盖四类工程问题：

**① 跨系统的端到端任务自动化**
- 痛点：一个完整任务需要串联多个系统（代码仓库→训练平台→指标系统→知识库），人工逐个操作、逐个填表，繁琐且易错。
- Meta-Agent 的解法：自然语言一句话驱动，自动编排全链路。**Skill（如 gm-cli 训练平台自动化）** 让智能体直接操作业务系统 API，实现从"分析需求→扫描代码→构建配置→执行→拉取结果→沉淀经验"的闭环。
- 实例：Gradmotion 训练平台上，创建一个训练任务需手填 20+ 个表单字段，ID 规则隐蔽（`goodsId` vs `goodsBackId` 易混），曾因 JSON 结构错误**连续失败 5 次**。接入 Skill 后，自动扫描 Git 仓库发现超参文件→dry-run 预检→一次创建成功。

**② 需要持续累积领域知识的工程任务**
- 痛点：老师的经验散落在聊天记录/脑子里，新人重复踩坑，项目知识无法复用。
- Meta-Agent 的解法：经验库 + 原则库 + 物理锚点库，自动召回相关经验注入上下文。

**③ 长周期、多轮迭代的优化任务**
- 痛点：参数搜索全靠手感，无法系统化、无法追溯、无法并行。
- Meta-Agent 的解法：Campaign 模式——DOE 参数采样 + 多保真度阶梯筛选 + 并行评估 + Pareto 多目标前沿 + 人工检查点。

**④ 需要安全自治的任务**
- 痛点：想让 AI 自己跑完一个复杂目标，但又怕失控、怕跑偏、怕删库。
- Meta-Agent 的解法：auto 模式——工作区硬监狱 + verify 完成校验 + drift 目标漂移检测 + 断路器 + 可中断/可恢复 checkpoint。

## 第 4 页 · 效果量化与受众规模

### 实际场景的效果对比

**场景一：跨系统端到端任务（以训练平台为例，Skill：gm-cli）**

| 指标 | 使用前（人工操作） | 使用后（Meta-Agent 驱动） | 提效 |
| --- | --- | --- | --- |
| 创建一个任务 | 查 ID + 填 20+ 字段 + 试错，**8–15 min** | 1 条指令 + 自动扫库 + dry-run 预检，**1–2 min** | **省 ~80–90%** |
| 任务首次成功率 | 曾连续失败 5 次 | dry-run + 软限制拦截，**首次即过** | 失败率大幅下降 |
| 日志跟踪 / 指标分析 | 手动刷新 + 截图对比 | `--follow` 自动跟踪 + 自动取数分析 | 人工值守→无人值守 |
| 经验复用 | 散落聊天/脑记 | 自动沉淀笔记 + 知识库召回 | 一次性→可累积资产 |

**场景二：知识沉淀的复利效应**

- 已沉淀的真实经验（示例）：
  - 建任务必须按 Skill 模板提交（源自一次连续失败 5 次的教训）
  - 算法设计前必须先读码算数再动手（源自 reward scale 偏大 2500× 的教训）
  - 算法开发先搜论文再写代码（源自重复造轮子的教训）
- 这些经验每次任务自动召回，**新人不再重复踩坑**。

### 受众与规模
- **直接使用者**：算法工程师 / 工程开发人员——把跨系统操作从"逐字段填表"升级为"自然语言指挥"。`【待确认：当前使用人数 / 覆盖的业务线】`

## 第 5 页 · 技术实现 ① 智能体框架（核心亮点）

### 运行环境
- **形态**：Node.js（≥18）CLI + 可编程 TypeScript SDK（npm 包 `@meta-agent/runtime`）。
- **部署**：本地终端运行；工作目录 fail-closed 沙箱（macOS sandbox-exec / Linux bwrap）。
- **多模型自动探测**：按环境变量优先级自动落到 Zhipu/GLM（默认）、DeepSeek、Qwen、Anthropic，统一封装各家差异。

### 自研运行时架构（项目结构 16 个模块）

```
kernel/    → 流式模型调用、工具循环、上下文压缩、权限、成本统计
core/      → 高层 Session、auto checkpoint/verify/drift 关卡、系统提示
routing/   → 模式检测 ModeDetector + SessionRouter（模式自动推断）
providers/ → 多提供商注册表（协议/计费/能力/探测）
tools/     → 内置 35 个工具（fs/shell/network/mcp/ui/system/agent/provenance）
subagent/  → 子代理调度、事件总线、委派工具
coordination/ → Campaign 并行评估、Pareto、胶囊、监控
campaign/  → Campaign 状态、插件框架、DOE
robotics/  → 硬件档案、知识三层、Team 协作
workflow/  → 阶段式工作流
validation/ → V&V Hook（量纲/单位/物理约束/OOM）
provenance/ → 数据溯源与血缘追踪
sandbox/   → OS 级沙箱
```

### 六种会话模式——一个运行时，按场景切换

| 模式 | 场景 | 关键能力 |
| --- | --- | --- |
| `agentic` | 通用工程任务 | 多轮工具循环、子代理委派、上下文压缩 |
| `auto` | 无人值守自治 | 工作区硬监狱、verify/drift 关卡、断路器、checkpoint 恢复 |
| `simple_auto` | 轻量无人值守 | 同款监狱，去掉自监督开销，面向短任务 |
| `auto_orch` | 多 Agent 自编排 | 执行节点 + 校验/航向/复核角色节点，编排即数据 |
| `campaign` | 长周期优化 | DOE、多保真度、并行评估、Pareto、人工检查点 |
| `robotics` | 机器人开发 | 硬件档案、三层知识库、Git 工作树并行、Team 协作 |

## 第 6 页 · 技术实现 ② 核心组件

| 维度 | 实际使用 | 说明 |
| --- | --- | --- |
| **底层大模型** | **GLM-5.2**（主，1M 上下文）+ **GLM-4.7**（回退）+ **GLM-4.5-Air**（flash 轻量判定） | 默认开启扩展思考；可平滑切换 DeepSeek / Qwen / Anthropic |
| **Skill / 工具** | 内置 **35 个工具** + **gm-cli Skill**（训练平台 34 个 API） | Skill 按需加载；工具经溯源 / 量纲校验插桩 |
| **MCP 集成** | 已接入 **web-search-prime**（联网搜索）；支持任意 MCP 服务器扩展 | `research_dispatch` 隔离调研，落盘报告，不污染主上下文 |
| **UI 交互** | **终端 CLI**（交互式对话 + 流式输出）+ 可编程 SDK | `Ctrl+G` 注入修正、`/experience` `/team` 等命令；非交互场景 `--yes` 自动批准 |
| **子代理编排** | 同步 `run_agent`（阻塞）+ 异步 `spawn_sub_agent`（并发扇出，默认 4） | 写操作强制 git 分支隔离，断路器限轮数/预算/时长 |
| **工程验证** | V&V Hook（量纲/单位/物理约束/OOM 检查）| 在工具调用前拦截不合理的工程计算 |
| **数据溯源** | provenance 溯源 + 血缘追踪 + 重复计算去重 | 每个关键结果可追溯完整来源链 |

### 子代理并发：把任务拆开并行跑

```
主代理 ──┬── spawn_sub_agent(子任务A)  ──┐
         ├── spawn_sub_agent(子任务B)  ──┤  并行执行，各自独立上下文
         └── run_agent(前置依赖任务)    ──┘  写操作在独立 git 分支，事后合并
                    │
              事件驱动通知完成 → 主代理收口合并
```

- **读隔离**：只读任务默认 `shared_readonly`，可放心并发。
- **写隔离**：写文件必须 `isolated_write`，独立 git 分支，主代理串行合并——**绝不并发写同一棵树**。

## 第 7 页 · 技术实现 ③ 知识库：怎么越用越懂、且不退化

### 四层知识体系（核心差异化：AI 提议，人来裁决）

| 知识库 | 记什么 | 更新纪律 |
| --- | --- | --- |
| **Experience 经验库** | 带置信分层（observed→reproduced→derived→reported→hypothesis）的结构化教训：问题、方案、被证伪假设、证据引用 | 任务中自动抽取 → **待审核队列** → `/experience review` 人工批准 → 入库；每轮自动召回 top-4 |
| **Principle 原则库** | 经验反复验证后晋升的可迁移原则，含第一性原理支撑 + 适用边界 + 不适用条件 | 晋升硬门槛：检索分 ≥ 450 且 ≥ 3 条经验收敛；**单独一条经验永远只是观测点，不会成为原则** |
| **Physical Anchor 物理锚点** | 模型不得推翻的硬件/物理事实（如"该关节力矩上限比额定低 15%"） | 带作用域（global/robot/code），记录观测/矛盾计数 |
| **Memory 记忆** | 用户偏好与协作规则（如"先读码算数再设计 reward"） | `memory_write` 提议 → `/memory review` 人工批准 → 跨会话持久 |

### 持续更新的闭环

```
physical anchor 钉死硬约束 → campaign 在约束下跑实验 → 产出 experience
     ↑                                                    ↓
     └── 反复验证后晋升为 principle（由 anchor 背书）←── 人工 review 把关
```

**一句话**：AI 负责高召回地提议候选，人负责高精度地把关。错误知识被挡在共享库门外，知识质量随时间复利式提升。

## 第 8 页 · 下一步计划 ①：自治模式的三条演进路线

### 路线一：auto 模式 → 微信远程干活

**目标**：让智能体成为你的"远程手脚"——你在微信发一句话，它在服务器上无人值守跑完，结果推送回来。

```
微信消息 "把构建跑绿，修掉所有失败用例"
    │
    ▼
auto 模式接管（工作区硬监狱 + verify/drift 关卡 + 断路器）
    │
    ├── 自动分析失败原因 → 逐个修复 → 跑测试验证
    ├── 进度写入 durable checkpoint（可中断/可恢复）
    └── 完成后推送结果到微信
```

- **价值**：碎片时间也能驱动工程任务。通勤路上一句话，到公司任务已经跑完。
- **技术基础已就绪**：auto 模式的硬监狱、verify/drift 关卡、checkpoint 恢复、失败重试均已实现，只需补一个微信消息网关层。

### 路线二：auto_orch 模式 → 承接 Loop Engineering

**目标**：面对复杂工程目标，智能体自主编排"计划→执行→校验→航向修正→复核"的完整循环。

- **编排即数据**：主代理先把复杂目标规划成一张"计划图"（执行节点干活 + 审查角色节点把关），固定引擎解释执行，非法编排自动回退到安全计划。
- **显式审查**：完成度校验、目标漂移检测、产出复核由计划图中的显式角色节点承担。
- **适用场景**："把整个数据管线重写为流式架构，并补齐单测与文档"这类需要拆解、并行、多轮验证的大任务。

### 路线三：/team 模式 → 个体（人 + Meta-Agent）与个体协作

**目标**：多个"人 + 智能体"单元组成团队，通过共享实验记录本协作完成大工程。

```
单元 A（算法工程师 + Meta-Agent）──┐
单元 B（测试工程师 + Meta-Agent）──┤── git 共享实验记录本（team.json）
单元 C（部署工程师 + Meta-Agent）──┘    乐观锁 + 显式抢占
```

- 每个"人 + Agent"单元在各自隔离的 Git 工作树中实验，通过共享的 `team.json` 看板协调目标、分工和进度。
- **本质**：不是"一个超级 Agent 什么都能干"，而是"每个工程师都有一个懂自己项目的智能体副驾，团队级别协同推进"。

1. **真实场景试点**：希望指定 1–2 个实际工程/算法项目作为试点，沉淀真实指标基线（任务数、迭代次数、人时节省），用数据验证提效结论。
